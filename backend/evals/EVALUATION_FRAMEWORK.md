# ARIA — Week 4 Evaluation Framework
*Mastering Agentic AI Certification — Gen Academy, "Evaluate Your Agent"*

This documents the Week 4 deliverable against ARIA, the Week 3 multi-agent
requirements-to-BRD/PR/Jira/Slack pipeline. Companion artifacts:
[METRICS.md](METRICS.md) (metric definitions), [golden_dataset.py](golden_dataset.py)
(the dataset), [evaluators.py](evaluators.py) (evaluator code),
[EVALUATION_REPORT.md](EVALUATION_REPORT.md) (the narrative failure-analysis report).

## The Primer (one-liner)

> I will measure **structured-output success, split correctness, entity
> correctness, extraction faithfulness, split fidelity, and p50/p90 latency**
> on **ARIA's Orchestrator agent** using a golden dataset of **25 cases**
> covering **single-requirement, multi-requirement, vague, entity-confusion,
> and edge-case scenarios**, with **code-based exact/keyword matching for 3
> metrics and same-provider LLM-as-judge for 2 metrics**. Pass bar: **≥95%
> structured-output success, ≥90% entity correctness, ≥0.85 extraction
> fidelity, p90 latency under 30s, cost not yet trackable**. I will run this
> in LangSmith and report the delta from **baseline (post-harness-fix run)**
> to **post-improvement (post-retry-budget-fix run)**.

**Honest framing up front**: ARIA has 5 stages (Orchestrator, Notion, GitHub,
Jira, Slack). This evaluation round covers **Orchestrator only**. The other 4
have LangSmith tracing but zero golden-dataset quality evaluation — see "What
is next."

## The Framework

| Field | Fill in |
|---|---|
| **Agent under test** | ARIA's Orchestrator stage — the LangGraph node that splits free-text input into individual requirements and extracts structured fields (title, description, acceptance criteria, priority) from each. Chosen over the other 4 stages because it's pure LLM (no side effects, safe to run repeatedly) and is where this project's own history shows a real bug class (confusing two similarly-worded requirements about different subjects). |
| **User outcome** | A PM/engineer pastes free-text requirements (a Slack message, meeting notes, a bug report) and gets back the *correct number* of distinct requirements, each attributed to the *correct subject*, faithfully describing what was actually asked — with no merging, no invented scope. If this is wrong, every downstream artifact (Notion BRD, GitHub tech doc, Jira ticket) is built on a wrong foundation, and nobody downstream double-checks it. |
| **Metrics (5)** | 1) `structured_output_success` — did parsing succeed without exhausting retries. 2) `split_count_match` — right number of split-out requirements. 3) `entity_keyword_match` — each requirement attributed to the right subject. 4) `extraction_fidelity` — title/description/criteria faithfully represent the source, 1-5 scale. 5) `split_fidelity` — the full split set covers all expected requirements with no merge/invention, 0-1 scale. Latency (p50/p90) and token count tracked as the paired cost/behavior metrics. |
| **Judge method** | `structured_output_success`, `split_count_match`, `entity_keyword_match` → **code-based** (status check / exact match / keyword-substring match). `extraction_fidelity`, `split_fidelity` → **LLM-as-judge**, using the *same* Mistral model as production (`mistral-large-latest`, temperature 0.0) with a structured Pydantic grading schema. **No human review and no independent (different-provider) judge were used this round** — flagged as a real gap below. |
| **Golden dataset** | 25 hand-authored cases (below the course's 30-50 recommended range — flagged as a gap), in [golden_dataset.py](golden_dataset.py). Scenario mix: single (5, 20%), multi-requirement (8, 32%), vague/underspecified (4, 16%), entity-confusion adversarial (3, 12%), edge cases — long dumps, contradictions, one-word input, filler speech (5, 20%). One case (`multi-01`) is a real production input reused verbatim from earlier debugging, not synthetic. **Labeling caveat**: all 25 cases and their expected labels (`expected_split_count`, `expected_requirements`) were authored by the AI assistant during this project, not independently hand-labeled or reviewed by a human domain expert before use — a deviation from the course's "hand-labeled by you or a domain expert" rule that should be corrected (a human pass over the labels) before treating these as fully trustworthy ground truth. |
| **Pass bar** | `structured_output_success` ≥95% · `entity_keyword_match` ≥90% (on successful extractions) · `split_count_match` ≥90% · `extraction_fidelity` ≥0.85 · `split_fidelity` ≥0.90 · latency p90 <30s for the orchestrator-only call · cost target not set (not yet measurable — see Instrumentation). **This bar was defined retroactively while writing this report — it did not exist when the evals were run**, which is itself a process gap worth naming. |
| **Instrumentation** | LangSmith tracing (`LANGCHAIN_TRACING_V2`, project `aria`) wired into **both** local/eval runs and the live Fly.io production deployment. Traced: full graph run → per-node run → per-LLM-call run, each with token usage (prompt/completion/total) and latency; custom metadata (`submission_id`, `requirement_index`, `requirement_count`) tagged on every call so a trace can be tied back to one specific input. Tool calls (MCP calls for Notion/GitHub/Jira) are traced for the other 4 stages but not graded. **Not captured**: per-run dollar cost — LangSmith's Mistral integration didn't return pricing metadata, so cost is currently visible only as raw token counts, not dollars. |
| **Baseline run** | Run "orchestrator-eval-2158c182" (the first run after fixing an eval-harness bug — see Failure analysis #3 below; the very first run before that fix scored nonsensically low and is excluded as invalid). Results: `structured_output_success` 71%, `entity_keyword_match` 71%, `split_count_match` 88%, `split_fidelity` 0.91, `extraction_fidelity` 0.67 (16/33 judge calls errored), latency p50 16.3s / p90 42.1s, avg 2,110 tokens/run. [LangSmith run](https://smith.langchain.com/o/c0324445-5648-41ac-8b3d-287d0b2e7b47/datasets/7d37b4c3-69c2-4750-801f-21f81634d736/compare?selectedSessions=7cacfc26-39b9-452d-bef4-17dbdcb5e488) |
| **Failure analysis** | **#1 (dominant, ~30% of cases): rate-limit-induced extraction fallback.** Under sustained Mistral API load, the extraction call's retry budget exhausted and the code's deliberate fallback path (mark the item "failed", return empty fields) kicked in. This is the single largest cost: every `structured_output_success` and `entity_keyword_match` miss traced back to this. **#2 (~10% of cases, same root cause): rate-limit-induced split fallback.** Same mechanism one level up — the split call falls back to treating multi-requirement input as one item (e.g. a 5-requirement case collapsed to `split_count: 1`). **#3 (eval-infrastructure bug, not a product bug): LLM-judge calls had no retry/timeout protection and the eval harness ran 3 examples concurrently** — production never runs more than one concurrent Mistral call, so this stacked more simultaneous load on a shared free-tier quota than the system is designed for, producing a first run with near-zero scores that didn't reflect product quality at all. **Notably absent**: zero cases of genuine entity confusion, requirement merging, or invented scope — the failure mode this dataset was specifically built to catch never appeared, in 75 example-executions across 3 runs. |
| **Improvement hypotheses (3-4)** | 1) **(Implemented)** Lower eval concurrency to 1 + add retry/timeout to judge calls — targets failure #3 — predicted: judge errors and harness-induced false failures drop to ~0. 2) **(Implemented)** Bump Orchestrator's own retry budget from `attempts=2` to `attempts=3` (timeout 25s→14s, still under the 60s node deadline) — targets failures #1/#2 — predicted: `structured_output_success`/`split_count_match` rise meaningfully. 3) **(Proposed, not yet tried)** Provision a separate Mistral API key for eval/dev traffic so it stops competing with the live deployment's quota — targets the root cause behind #1/#2 directly, since the in-process retry budget can mitigate but can't fully absorb contention from a process it has no visibility into. 4) **(Proposed, not yet tried)** Independent (different-provider) LLM-as-judge cross-check on a sample, to calibrate the same-provider judge per the course's human/judge-agreement guidance. |
| **Post-improvement run** | Run "orchestrator-eval-f2b06ff5". Results vs baseline: `structured_output_success` 68% (**-3pp, essentially flat/noise**), `entity_keyword_match` 68% (**-3pp**), `split_count_match` 92% (**+4pp**), `split_fidelity` 0.95 (**+0.04**), `extraction_fidelity` 0.60 with error rate improving 16/33→8/29 (**-20pp errors**), latency p50 20.4s / p90 51.6s (**worse, +4.1s/+9.5s** — more retry attempts cost time), avg 2,191 tokens/run (**+4%, noise-level**). [LangSmith run](https://smith.langchain.com/o/c0324445-5648-41ac-8b3d-287d0b2e7b47/datasets/7d37b4c3-69c2-4750-801f-21f81634d736/compare?selectedSessions=bcea4de7-7f99-4985-98ee-e936e5c590d8) · **Verdict against pass bar**: split_count_match and split_fidelity now pass; structured_output_success, entity_keyword_match, extraction_fidelity, and p90 latency still fail the bar — driven by the same ambient quota-contention issue that hypothesis #3 (not yet implemented) targets directly. |
| **What is next** | Top remaining failure mode: Mistral API quota shared between eval traffic and the live Fly deployment — the retry-budget fix helped the metrics it could reach but can't eliminate contention it has no visibility into. With another week: (1) separate API key for eval traffic, (2) build the curated full-pipeline subset to extend evaluation to Notion/GitHub/Jira/Slack (currently zero quality coverage on those 4 stages), (3) get a human pass over the golden dataset's labels per the caveat above, (4) wire cost tracking (manual token×price calc, since LangSmith isn't returning it automatically), (5) wire this as a recurring CI gate instead of a manual on-demand run. Production monitoring strategy: alert if `structured_output_success` (sampled from live traces, not just eval) drops >10pp week-over-week, or if p95 latency exceeds the existing 60s node deadline on >5% of runs. |
